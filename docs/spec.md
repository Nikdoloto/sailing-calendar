## Пользователи и роли

В MVP есть четыре заранее созданных профиля:

- Никита, роль `admin`
- Даня, роль `participant`
- Лиза, роль `participant`
- Настя, роль `participant`

Регистрации нет. Никита - единственный пользователь, который может бронировать тренировку.

## Основные сценарии

### Вход

1. Пользователь открывает приложение.
2. Выбирает профиль.
3. Вводит PIN или пароль.
4. Попадает в календарь.
5. Повторный вход не требуется, пока активна cookie-сессия.

### Отметка доступности

1. Пользователь открывает календарь месяца.
2. Видит выделенные пятницу, субботу и воскресенье.
3. Открывает дату.
4. Отмечает `Утро`, `День`, `Вечер` или `Весь день`.
5. Видит, кто из команды уже свободен в каждом слоте.

### Подбор даты

1. Команда отмечает доступность.
2. Календарь подсвечивает даты:
   - зелёный: есть слот, где свободны все 4;
   - жёлтый: есть слот, где свободны 3 из 4;
   - нейтральный: меньше 3.
3. Никита выбирает подходящую дату и слот.

### Бронирование

1. Никита открывает дату.
2. Заполняет слот, инструктора и комментарий.
3. Сохраняет тренировку.
4. Все участники видят статус `Тренировка забронирована`.

## Критерии приемки

- [ ] Пользователь может войти только под одним из четырёх профилей.
- [ ] Неверный PIN или пароль не пускает в приложение.
- [ ] После входа приложение помнит пользователя через cookie-сессию.
- [ ] Календарь месяца работает на мобильной ширине экрана.
- [ ] Пятница, суббота и воскресенье визуально выделены.
- [ ] Пользователь может отмечать и снимать свою доступность по слотам.
- [ ] `Весь день` включает или выключает `Утро`, `День`, `Вечер`.
- [ ] В каждом слоте видно, кто из участников свободен.
- [ ] Дата подсвечивается зелёным, если есть слот со всеми 4 участниками.
- [ ] Дата подсвечивается жёлтым, если есть слот с 3 из 4 и нет слота со всеми 4.
- [ ] Никита может создать или обновить бронирование тренировки.
- [ ] Даня, Лиза и Настя не могут создать или обновить бронирование.
- [ ] Забронированная тренировка видна всем участникам.
- [ ] Приложение содержит PWA manifest и service worker.
- [ ] Дизайн использует токены из дизайн-системы.

## Границы MVP

Не делаем:

- регистрацию;
- восстановление пароля;
- интеграцию с внешними календарями;
- уведомления;
- сложные роли;
- расширенные личные профили.

## Допущения

- Для локального MVP используются демо-PIN: `1111`, `2222`, `3333`, `4444`. Перед реальным запуском их нужно заменить через переменные окружения.
- Сервер запускается на Node.js 24+, потому что используется встроенный модуль `node:sqlite`.
- Время приложения хранится в формате даты `YYYY-MM-DD`; часовой пояс для людей - Москва.
- Сессия в MVP живёт 90 дней.
- `Весь день` не хранится как отдельный слот, а включает или выключает `Утро`, `День`, `Вечер`.
- Бронирование не требует автоматической проверки, что свободны все 4 или 3 из 4. Никита может принять решение вручную.
- Если доступность участников изменилась после бронирования, бронь остаётся на дате, пока Никита её не изменит или не снимет.
- В MVP есть ручной выход из аккаунта.
- PWA поддерживает установку и базовый cache статических файлов, но не offline-редактирование данных.
